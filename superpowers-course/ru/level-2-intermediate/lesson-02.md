# Спецификации и планы

## Введение

Как создавать качественные спецификации и планы реализации.

## Спецификация

Спецификация — это результат brainstorming-фазы. Хорошая спецификация содержит:

- **Цель** — что мы делаем и зачем
- **Требования** — что система должна делать
- **Ограничения** — что система НЕ должна делать
- **Интерфейс** — как пользователь взаимодействует

### Советы

- Показывайте спецификацию по частям — не пытайтесь утвердить всё сразу
- Задавайте вопросы к каждой части
- Не стесняйтесь просить изменения

## План реализации

План создаётся навыком `writing-plans`. Каждая задача содержит:

- Описание (1-2 предложения)
- Файлы для изменения (точные пути)
- Код (полностью, не заглушки)
- Шаги проверки

### Правила хорошего плана

- Задачи по **2-5 минут**
- Каждая задача **самодостаточна**
- Порядок задач учитывает **зависимости**
- Следует принципам **YAGNI** и **DRY**

## Пример задачи из плана

```markdown
### Задача 3: Эндпоинт создания заметки

**Файлы:**
- test/routes/notes.test.ts
- src/routes/notes.ts

**Действия:**
1. Написать тест POST /notes
2. Убедиться что тест падает
3. Реализовать обработчик
4. Убедиться что тест проходит

**Проверка:** npm test — все тесты зелёные
```

## Итоги

- Спецификация определяет ЧТО делать
- План определяет КАК делать
- Задачи должны быть маленькими и самодостаточными
- YAGNI и DRY — ключевые принципы
